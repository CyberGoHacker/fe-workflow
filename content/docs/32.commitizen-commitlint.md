# Commitizen & Commitlint




## 1 commitizen & cz-conventional-changelog 规范提交信息格式


#### 1.1 安装、配置依赖

```bash
# git cz 替代 git commit，生成符合规范的 commit message。
pnpm add commitizen -D

# 一个符合 Angular 团队规范的 preset，按照我们指定的规范生成 commit message。
pnpm add cz-conventional-changelog -D


# 【全局模式】下, 需要 ~/.czrc 配置文件, 为 commitizen 指定 Adapter。
echo '{ "path": "cz-conventional-changelog" }' > ~/.czrc

# 【项目级配置】下
# package.json
{
  "script": {
    "commit": "git-cz",
  },
  "config": {
    "commitizen": {
      "path": "node_modules/cz-conventional-changelog"
    }
  }
}

# ===> 或者使用命令代替
npm set-script commit "git-cz"
npx commitizen init cz-conventional-changelog --save-dev --save-exact --force
```


#### 1.2 自定义 Adapter

```bash
# 指定一套符合自己团队的规范代替 Angular 规范
pnpm add cz-customizable -D

# package.json 指定配置
{
  "config": {
    "commitizen": {
      "path": "node_modules/cz-customizable"
    }
  }
}
# ===> 或者配置 .czrc 文件
{ "path": "cz-customizable" }
```


#### 1.3 配置 .cz-config.js

```js
'use strict'
module.exports = {
  // type 类型
  types: [
    { value: '✨feat', name: 'feat: 新增内容' },
    { value: '🐛fix', name: 'fix: 修复Bug' },
    { value: '📝docs', name: 'docs: 文档变更' },
    { value: '💄style', name: 'style: 代码格式（不影响功能，例如空格、分号等格式修正）' },
    { value: '♻️refactor', name: 'refactor: 代码重构（不包括 bug 修复、功能新增）' },
    { value: '⚡️perf', name: 'perf: 性能优化' },
    { value: '✅test', name: 'test: 添加、修改测试用例' },
    {
      value: '📦️build',
      name: 'build: 构建流程、外部依赖变更（如升级 npm 包、修改 webpack 配置等）',
    },
    { value: '🔨ci', name: 'ci: 修改 CI 配置、脚本' },
    { value: '🔧chore', name: 'chore: 对构建过程或辅助工具和库的更改（不影响源文件、测试用例）' },
    { value: '⏪revert', name: 'revert: 回滚 commit' }
  ],

  // 针对每一个 type 去定义对应的 scopes，例如 fix
  // scopeOverrides: {
  //   fix: [
  //     {name: 'merge'},
  //     {name: 'style'},
  //     {name: 'e2eTest'},
  //     {name: 'unitTest'}
  //   ]
  // },

  // scope 类型
  scopes: [
    ['view', '页面相关'],
    ['components', '组件相关'],
    ['utils', 'utils 相关'],
    ['styles', '样式相关'],
    ['deps', '项目依赖'],
    ['workflows', '部署相关'],
    ['md', '文档相关'],
    ['other', '其他修改'],
    // 如果选择 custom，后面会让你再输入一个自定义的 scope。也可以不设置此项，把后面的 allowCustomScopes 设置为 true
    ['custom', '以上都不是？我要自定义'],
  ].map(([value, description]) => {
    return {
      value,
      name: `${value.padEnd(30)} (${description})`,
    }
  }),

  // 是否允许自定义填写 scope，在 scope 选择的时候，会有 empty 和 custom 可以选择。
  // allowCustomScopes: true,

  // allowTicketNumber: false,
  // isTicketNumberRequired: false,
  // ticketNumberPrefix: 'TICKET-',
  // ticketNumberRegExp: '\\d{1,5}',

  // 交互提示信息
  messages: {
    type: '确保本次提交遵循 Angular 规范！\n选择你要提交的类型：',
    scope: '\n选择一个 scope（可选）：',
    customScope: '请输入自定义的 scope：',
    subject: '填写简短精炼的变更描述：\n',
    body: '填写更加详细的变更描述（可选）。使用 "|" 换行：\n',
    breaking: '列举非兼容性重大的变更（可选）：\n',
    footer: '列举出所有变更的 ISSUES CLOSED（可选）。 例如: #1, #2：\n',
    confirmCommit: '确认提交？',
  },

  // 设置只有 type 选择了 feat 或 fix，才询问 breaking message
  allowBreakingChanges: ['feat', 'fix'],

  // 跳过要询问的步骤
  // skipQuestions: ['body', 'footer'],

  // subject 限制长度
  subjectLimit: 100,
  breaklineChar: '|', // 支持 body 和 footer
  // footerPrefix : 'ISSUES CLOSED:'
  // askForBreakingChangeFirst : true,
}
```




## 2 commitlint 校验提交信息

```bash
# lint commit messages
pnpm add @commitlint/cli -D

# 一份符合 Angular 团队规范校验的配置
pnpm add @commitlint/config-conventional -D

# 对自定义的 Adapter 进行检测
pnpm add commitlint-config-cz -D
```


#### 2.1 配置 .commitlintrc

```json
{
  "extends": [
    - "@commitlint/config-conventional" // 默认 commitizen adapter
    + "cz" // 自定义 commitizen adapter
  ],
  "rules": {}
}
```




## 3 结合 Husky

```json
// package.json
{
  "husky": {
    "hooks": {
      "commit-msg": "commitlint -e $GIT_PARAMS"
    }
  }
}

// ===> 或者用命令代替
npx husky add .husky/commit-msg 'npx --no-install commitlint --edit "$1"'
```
